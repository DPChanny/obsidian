# Gemini CLI 지시 사항

이 파일은 이 보관소 내에서 Gemini CLI가 수행하는 모든 작업에 적용되는 최상위 지침입니다.

## 1. 보관소의 목적 및 성격

- **지식 베이스**: 단순 요약이 아닌 기술적 개념의 **철학, 배경, 동작 원리**를 깊이 있게 탐구하는 저장소입니다.
- **연결된 지식**: 각 개념은 고립되지 않고 폴더 또는 관련 노트와 상호 연결되어야 합니다.

## 2. 문서 작성 가이드라인

- **언어**: 한국어를 기본 언어로 사용하되, 기술 용어는 영어 원문을 병기하거나 영문 그대로 사용함.
- **문체**: 서술형 문장 대신 **'~함', '~임'** 등 명사형/개조식으로 간결하게 종결함.
- **시각 요소**: 아이콘이나 이모지를 일절 사용하지 않고, 텍스트와 코드 블록 위주로 정결하게 작성함.
- **제목 구성**: 파일 최상단에 파일명과 중복되는 별도의 제목(H1 등)을 작성하지 않음. 파일명 자체가 문서의 제목 역할을 하도록 함.
- **코드 블록**: 반드시 해당 언어(예: `cpp`, `tsx`, `json`)를 명시하며, 주석을 통해 로직을 설명함.
- **수식**: 수학적 표현이 필요한 경우 LaTeX($ ... $)를 사용함.

## 3. 링크 및 참조 규칙

- **인라인 Wikilinks**: 본문 내 키워드가 이미 보관소 내에 노트로 존재한다면, 해당 키워드에 직접 `[[파일명|표시텍스트]]` 형식을 사용하여 연결합니다.
- **`related` 필드 활용 (Frontmatter)**:
  - 본문에 직접적으로 언급되지 않았으나, 맥락상 깊이 연관된 개념은 YAML Frontmatter의 `related` 필드에 추가합니다.
  - **DRY 원칙**: 본문 내에 이미 인라인 링크로 포함된 문서는 `related` 필드에 중복하여 기재하지 않습니다.
- **상호 연결성**: 새로운 노트를 작성하거나 기존 노트를 수정할 때, 관련된 `Generals/` 개념이나 상위/하위 기술 스택과의 연결을 항상 고려합니다.

## 4. 디렉토리 구조 및 명명 규칙

### 4.1 분류 체계 원칙

1. **복수형 집합 (Plural)**: 여러 하위 카테고리를 포함하는 대분류 또는 중분류는 복수형으로 명명합니다. (예: `Languages`, `Frameworks`, `Tools`)
2. **단수형 대표 명사 (Singular)**: 특정 개념이나 기술 스택의 시작점, 또는 독립적인 기술 집합은 단수형 대표 명사를 사용합니다. (예: `Git`, `Anaconda`, `Network`, `OOP`)
3. **`Generals`의 활용**: 불필요한 중첩 폴더를 피하기 위해, 어떤 복수형 폴더 하위에도 해당 범주의 범용적 개념을 담는 `Generals/` 폴더를 자유롭게 생성할 수 있습니다. (예: `Languages/Generals/`)
4. **계층적 구조 준수**: 루트 디렉토리(`Academics`, `Devs`, `Generals`, `Templates`)의 분류 체계를 엄격히 따릅니다.

### 4.2 주요 디렉토리 설명

- **`Academics/`**: 순수 학문적 지식 및 이론 (논문 요약 등).
- **`Devs/`**: 실무 개발과 직결된 기술 스택. (`Frameworks`, `Languages`, `Libraries`, `Tools` 포함)
- **`Generals/`**: (Root 하위) 특정 기술에 종속되지 않는 컴퓨터 과학의 근본 원리 및 전 영역을 관통하는 범용 지식.
- **`Templates/`**: 노트 작성을 위한 표준 서식 및 로그 템플릿.

## 5. Gemini CLI 작업 원칙

- **기존 맥락 유지**: 새로운 내용을 추가할 때(`obsidian_patch_content`) 기존의 논조와 서술 스타일을 파괴하지 않음.
- **중복 방지**: 이미 존재하는 개념은 새로 노트를 만들기보다 기존 노트를 확장하거나 링크로 대체함.
- **출처 표기**: 외부 자료(블로그 등)를 참고한 경우 링크를 포함함.

## 6. 학습 프로세스 기록 및 문서화

- **학습 기록 시점**: 사용자의 명시적인 **'정리해줘'**, **'기록해줘'** 등의 명령이 내려졌을 때만 Obsidian에 기록함. 세션 종료 시점에 임의로 기록하지 않음.
- **유동적 포맷**: 대화의 성격에 따라 기록 형식을 최적화함.
  - **단순 개념 학습**: 개념(Concept), 실무 예시(Example), 핵심 요약(Summary) 위주로 구성함.
  - **심화 추론 및 토론**: 긴 논의가 있었던 경우, 논리적 전개 과정(Reasoning Steps), 주요 질문(Key Questions), 그리고 해결된 Q&A 섹션을 포함하여 기록함.
- **맥락 보존**: 단순히 결과물만 남기는 것이 아니라, 어떤 고민과 질문을 통해 해당 결론에 도달했는지의 '흐름'을 문서에 반영함.
