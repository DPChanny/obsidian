# Gemini CLI 지시 사항

본 저장소 내 Gemini CLI 작업에 적용되는 최상위 지침임.

## 1. 학습 (학습 및 대화 방식)

- **BUFFER.md (대화 보조 버퍼) 활용**:
  - 터미널 환경에서 렌더링하기 어려운 복잡한 수식이나 시각적 확인이 필요한 데이터를 사용자가 Obsidian을 통해 쉽게 확인할 수 있도록 대화 중 보조 수단으로만 사용함.
  - 임의로 파일이나 기존 내용을 삭제하거나 덮어쓰지 않으며, 항상 새로운 내용을 추가(Append)하는 방식으로만 기록함.
- **기록 대기**: 사용자의 명시적 명령('정리해줘', '기록해줘' 등)이 있기 전까지는 내부적으로 정보와 사고 흐름을 유지하며 대기함.
- **대화 스타일**:
  - 비유적 설명이나 은유적 표현을 엄격히 금지함.
  - 해당 분야의 정확한 전문 용어(Technical Terms)를 우선적으로 사용함.
  - 불필요한 수식어를 배제하고, 사실과 논리에 기반한 객관적인 문체로 답변함.

## 2. 정리 (문서 작성 및 Obsidian 규칙)

- **전용 도구 사용**: 일반 파일 시스템 도구(예: write_file, replace, grep_search 등) 사용을 엄격히 금지하며, 반드시 **Obsidian MCP 도구**만을 사용함.
- **문서 작성 규격**:
  - 언어: 한국어 기본, 기술 용어는 영어 원문 병기 또는 단독 사용함.
  - 문체: '~함', '~임' 등 명사형/개조식 종결 어미를 사용함.
  - 시각 요소: 이모지 및 아이콘 사용을 금지하며, 텍스트와 코드 블록 위주로 정결하게 작성함.
  - 제목 구성: 파일 최상단에 H1 제목 작성을 금지하며, 파일명을 문서 제목으로 간주함.
  - 코드 블록: 언어 명시(예: `cpp`, `tsx`) 및 주석을 통한 로직 설명을 필수로 함.
  - 수식: LaTeX($ ... $) 형식을 사용함.
- **문서 구조 최적화**:
  - 개념 학습: 개념(Concept), 실무 예시(Example), 요약(Summary) 중심.
  - 심화 토론: 논리 전개(Reasoning), 주요 질문(Key Questions), Q&A 중심.
  - 맥락 보존: 결론에 도달하기까지의 사고 흐름을 문서에 반영함.
- **링크 및 참조 (Wikilinks)**:
  - 기존 노트 언급 시 `[[폴더/파일명|파일명]]` 형식을 사용하며, 항상 절대 경로를 포함함.
- **`related` 필드 (Frontmatter)**:
  - 본문에 언급되지 않았으나 맥락상 연관된 개념을 추가함.
  - 절대 경로 및 파일명 필수: `[[폴더/파일명|파일명]]` 형식을 사용함.
  - 따옴표 필수: 파싱 오류 방지를 위해 반드시 큰따옴표로 감쌈. (예: `- "[[Devs/C/MSVC|MSVC]]"`)
  - DRY 원칙: 본문 내 인라인 링크와 중복 기재를 금함.
- **작업 원칙**:
  - 기존 문서의 논조와 서술 스타일을 보존함.
  - 기존 노트를 확장하거나 링크로 대체하는 것을 우선하여 중복을 방지함.
  - 외부 자료 참고 시 출처 링크를 포함함.

## 3. 저장 (폴더 및 디렉토리 구조 규칙)

- **명명 규칙**:
  - 복수형(Plural): 하위 카테고리를 포함하는 대/중분류에 사용함. (예: `Languages`, `Frameworks`)
  - 단수형(Singular): 특정 기술 스택의 시작점 또는 독립적 기술 집합에 사용함. (예: `Git`, `Network`)
- **디렉토리 체계**:
  - 계층 구조: 루트 디렉토리(`Academics`, `Devs`, `Generals`, `Templates`)의 분류 체계를 엄격히 준수함.
  - `Generals/`: 범주별 범용 개념을 담는 폴더로, 각 계층에서 자유롭게 생성 가능함.
- **보관소 원칙**:
  - 기술적 개념의 철학, 배경, 동작 원리를 깊이 있게 탐구함.
  - 모든 개념은 관련 노트와 상호 연결되어야 함.
